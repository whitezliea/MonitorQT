## 4. 逻辑一致性检查

- `MonitoringRuntimeService::acknowledgeAlarm()` 只在 `AlarmService::acknowledge()` 成功后发布确认事件，失败时返回错误并不发布事件，避免误写日志。
- `AlarmService` 仍保持领域状态变更职责，事件发布放在 runtime 应用服务层，符合既有事件链路分层。
- `OperationLogQuery` 新字段追加在结构体尾部，保留既有聚合初始化调用的字段顺序兼容性。
- `SQLiteOperationLogRepository::query()` 保留原 `maxCount` 语义；分页能力通过新增 `queryPage()` 提供，避免改变既有调用方行为。
- 分页查询使用与原查询一致的时间、等级、分类过滤条件，并额外返回 `totalCount`，便于 UI 或服务层做页码判断。
- Alarm Center 和 History 页面分页时会 clamp 当前页，数据不足一页或过滤结果为空时不会越界。
- Logs & Settings 的日志筛选在当前快照内完成，不直接让 UI 访问数据库，保持当前表现层通过快照刷新页面的设计。

## 5. 仍需后续闭环的边界

以下问题在本轮未作为短修复强行改造，原因是它们涉及主窗口数据源、组合根、后台 runtime、查询服务和配置仓储的整体接入：

1. `MainWindow` 当前仍消费 `UiSnapshotProvider` 演示快照源，尚未切换为真实 `MonitoringRuntimeService`、Worker 和 SQLite 查询服务的完整运行闭环。
2. Alarm Center 历史区和 History 页面已补分页 UI，但仍基于当前快照内存数据；后续应接入应用层查询服务，再由仓储执行 SQLite 时间范围、状态、等级和分页查询。
3. Logs & Settings 已补日志筛选 UI，但配置保存仍进入 `UiSnapshotProvider` 内存状态；后续应通过应用层服务接入 `SQLiteConfigurationRepository`，实现重启后可重载。
4. 数据源超时 Offline 到 UI 的完整验证仍依赖真实 runtime/UI 组合接入；当前 runtime 侧能力和测试仍保持通过。

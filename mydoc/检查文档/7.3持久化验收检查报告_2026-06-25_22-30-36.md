## 1. 总体结论

7.3 持久化核心能力基本完成，编译和现有自动化测试均通过。当前工程已经具备：

- SQLite schema version 为 5。
- SQLite WAL 已启用。
- `history_samples`、`alarm_events`、`operation_logs`、`tag_runtime_settings`、`runtime_settings`、`schema_migrations` 表由 migration 创建。
- 历史采样可按 Tag 配置的 `isHistorized`、`historyIntervalMs`、质量变化和警报状态变化入队。
- 历史、警报仓储支持分页查询；日志仓储支持时间、级别、分类过滤和 `maxCount` 限制查询。
- 停止采集时可显式 flush 历史队列。
- Worker 停止/析构时会 drain queue 并执行 shutdown flush。
- 历史保留任务可按 30 天策略批量清理旧数据，并写入操作日志。
- Tag 运行配置和 runtime settings 可保存并重新加载。

需要注意：`SQLiteOperationLogRepository::query()` 当前不是 `page/pageSize/totalCount` 形式的分页查询，而是 `maxCount` 限制查询，因此“日志查询支持分页”只能判定为部分通过。另一个边界是，Worker/Coordinator 层已具备应用退出 stop + flush 语义，但当前 `MainWindow` 仍使用演示 `UiSnapshotProvider`，未接入真实持久化 runtime；完整应用退出链路需要在 runtime 组合根接入后复验。

## 4. 逻辑一致性检查

- SQLite 连接一致：每次打开连接使用唯一 connection name，避免跨线程复用同一个 `QSqlDatabase`。
- 迁移一致：schema migration 版本连续，从 1 到 5；每个 migration 在事务中执行并写入 `schema_migrations`。
- 写入一致：历史、警报、操作日志、配置保存均使用事务；失败时 rollback。
- 查询一致：历史和警报查询校验 UTC 时间范围、page/pageSize，并返回 totalCount；警报支持 tag、level、state 过滤。
- 去重一致：历史表有 `idx_history_samples_tag_source` 唯一索引，仓储使用 `INSERT OR IGNORE` 避免同一 Tag/source 重复写入。
- 配置一致：`tag_runtime_settings` 保存告警阈值、历史开关、历史周期；`runtime_settings` 保存运行参数键值。
- 保留一致：30 天保留策略来自 runtime options 默认值，清理时分批删除，且记录操作日志。

## 5. 后续建议

1. 为 `operation_logs` 增加真正分页查询模型，例如 `OperationLogQueryResult { items, totalCount, page, pageSize }`。
2. 在 runtime 组合根中接入真实 `BatchPersistWorker` 和 `PersistenceRuntimeCoordinator`，并增加应用退出级别的 stop + flush 验收。
3. 为 `HistoryRuntimeStateConsumer` 增加配置刷新入口，确保保存后的 Tag 历史策略可在运行期同步到历史入队判断。

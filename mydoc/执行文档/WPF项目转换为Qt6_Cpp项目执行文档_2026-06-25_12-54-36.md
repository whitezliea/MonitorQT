目标项目：当前 `MonitorQT`，Qt6 / C++ / CMake  
本文目标：基于已有 C# 代码和 README，给出 WPF 项目迁移到 Qt6/C++ 的架构设计、模块参考关系、转换顺序和验收步骤。本阶段不写业务代码，只定义可执行的迁移路线。

## 1. 迁移目标与边界

### 1.1 目标

将 `MyProject` 中已经完成的多通道测量仪器监控 Demo，迁移为 Qt6/C++ 桌面项目。迁移后应保留当前 C# 项目的核心业务链路：

1. 模拟器按周期生成原始设备帧。
2. 原始帧经过数据清洗、Tag 映射、派生量计算和矩阵统计。
3. 警报服务基于质量状态和阈值产生运行时状态与警报生命周期事件。
4. 实时缓存、历史采样、警报事件、操作日志和矩阵分析通过事件消费者解耦。
5. 后台 Worker 批量写入 SQLite。
6. UI 每个刷新周期读取统一快照，而不是直接读设备或数据库。
7. Dashboard、Realtime Tags、Trend、Alarm Center、History、Measurement Map、Logs & Settings 页面功能保持一致。

### 1.2 非目标

本执行文档不要求本次直接写 Qt/C++ 源码。  
不在第一轮迁移中扩展真实 Modbus、串口、TCP、厂商 SDK、用户认证、网络 API 或生产安装器。  
不建议把 WPF XAML 逐行翻译成 Qt UI。应优先迁移业务模型、运行时链路和页面行为，再实现 Qt 页面。

### 1.3 迁移原则

1. 先核心链路，后页面细节。
2. 先领域模型和 Application 服务，后 SQLite、模拟器、UI。
3. 保持 `Domain` 纯净，不依赖 `QObject`、`QWidget`、SQLite、日志库或 UI。
4. Qt 表现层只消费 Application 层 DTO 或快照，不直接依赖模拟器和数据库细节。
5. 数据时间在领域和持久化中继续使用 UTC 契约。
6. C# 测试用例是 Qt 迁移验收标准的重要来源。

### 3.1 CMake 目标建议

建议逐步将当前单一可执行目标拆为多个静态库或对象库，依赖方向保持清晰：

```text
MonitorDomain
  <- MonitorApplication
      <- MonitorInfrastructure
      <- MonitorSimulator
      <- MonitorPresentation
          <- MonitorQT executable
```

建议 Qt 模块：

| Qt 模块 | 用途 |
| --- | --- |
| `Qt6::Core` | 时间、容器、线程、信号槽、对象生命周期 |
| `Qt6::Widgets` | 当前 QWidget 页面和主窗口 |
| `Qt6::Sql` | SQLite 仓储实现 |
| `Qt6::Charts` 或 QCustomPlot/Qwt | 趋势曲线替代 ScottPlot |
| `Qt6::Concurrent` | 可选，用于轻量后台任务 |
| `Qt6::Test` | Qt Test 单元测试 |

如果保持当前 `CMAKE_CXX_STANDARD 17`，所有设计都可以落地。若后续允许 C++20，可考虑用 `std::span`、`std::jthread` 和更清晰的时间 API，但不是迁移前置条件。

### 3.2 分层依赖规则

| 目标层 | 允许依赖 | 禁止依赖 |
| --- | --- | --- |
| `domain` | C++ STL，少量 QtCore 值类型需谨慎 | `QObject`、`QWidget`、SQLite、日志、模拟器、UI |
| `application` | `domain`、QtCore 或 STL 并发工具、日志抽象 | Qt Widgets、具体 SQLite 实现 |
| `infrastructure` | `application` 抽象、QtSql、日志实现、文件系统 | Qt 页面和 ViewModel |
| `simulator` | `domain`、`application` 数据源抽象 | Qt Widgets、SQLite |
| `presentation` | `application` DTO/usecase/service、Qt Widgets | 直接操作数据库表或模拟器内部对象 |
| `bootstrap` | 所有运行时模块 | 不承载业务逻辑 |

## 4. C# 到 Qt/C++ 的通用转换规则

| C# 概念 | Qt/C++ 建议 | 说明 |
| --- | --- | --- |
| `record` | `struct` 或值语义 `class` | DTO 和领域值对象优先用不可变或少修改的值对象 |
| `enum` | `enum class` | 避免命名污染，必要时提供字符串转换函数 |
| `Guid` | `QUuid` | SQLite 持久化时统一存字符串 |
| `DateTime` / `DateTimeOffset` | `QDateTime` 或 `std::chrono` | 统一 UTC，UI 显示时转本地时间 |
| `double?` / `string?` | `std::optional<double>` / `std::optional<QString>` | 表达缺失值，不用魔法值 |
| `IReadOnlyList<T>` | `std::vector<T>` 或 `QVector<T>` | 领域层建议 `std::vector`，UI 层可用 Qt 容器 |
| `Dictionary<string,T>` | `std::unordered_map<std::string,T>` 或 `QHash<QString,T>` | TagId 查找需大小写策略一致 |
| `Task` / `async` | `QThread`、`QtConcurrent`、`QTimer`、回调 | 运行时采集建议独立 worker thread |
| `CancellationToken` | `std::atomic_bool`、`QAtomicInteger` 或 worker stop flag | 所有循环必须可停止 |
| `Channel<T>` | 自定义 `BlockingQueue<T>` | 可用 `QMutex` + `QWaitCondition` + `QQueue` |
| `event` | Qt signal/slot 或纯 C++ EventBus | Application 层可保持非 UI EventBus |
| `[ObservableProperty]` | `Q_PROPERTY` + `NOTIFY`，或页面 `refresh(dto)` | 复杂页面优先快照刷新，减少属性风暴 |
| `[RelayCommand]` | Qt slot 或 `QAction`/按钮连接 | 命令入口转为槽函数 |
| WPF Dispatcher | `QMetaObject::invokeMethod(..., Qt::QueuedConnection)` | 所有 UI 更新回主线程 |
| ScottPlot.WPF | Qt Charts、QCustomPlot 或自绘 `QWidget` | 热力图建议 `QPainter` 自绘，趋势可用 Qt Charts |
| Serilog | `QLoggingCategory`、spdlog 或自定义滚动日志 | `operation_logs` 仍走 SQLite 业务日志 |

## 6. 分阶段转换步骤

### 阶段 0：冻结源项目行为和迁移清单

参考模块：

- `MyProject/README.md`
- `MyProject/Tests/`
- `MyProject/Domain/`
- `MyProject/Application/`
- `MyProject/MultiChannelMonitor/ViewModels/`
- 当前 Qt 骨架：`mainwindow.*`、`shell/`、`navigation/`、`pages/`

执行步骤：

1. 梳理 WPF 页面功能清单和运行链路，形成 parity checklist。
2. 按 C# 测试用例分类，标记哪些测试必须在 Qt 版复刻。
3. 固定默认参数：500 ms 采集、1 s UI 刷新、5 s 批量写入、30 天保留、1/5/30 分钟趋势窗口。
4. 确认 Qt 项目采用 Widgets 路线，当前 `QMainWindow + QStackedWidget + shell/nav/pages` 作为基础壳。

验收标准：

- 有明确模块映射表。
- 有页面功能清单。
- 有最小验收数据链路：模拟一帧 RawFrame，能在后续阶段验证 Tag、警报、矩阵和历史采样结果。

### 阶段 1：Qt 工程和组合根基础

参考模块：

- `MultiChannelMonitor/Bootstrap/RuntimeComposition.cs`
- `MultiChannelMonitor/Bootstrap/RuntimeCompositionDependencies.cs`
- `MultiChannelMonitor/Bootstrap/EventRegistration.cs`
- 当前 Qt `CMakeLists.txt`

执行步骤：

1. 在 CMake 中拆出 `MonitorDomain`、`MonitorApplication`、`MonitorInfrastructure`、`MonitorSimulator`、`MonitorPresentation`。
2. 补充 Qt 依赖：`Core`、`Widgets`、`Sql`，趋势图阶段再加入 `Charts` 或第三方绘图库。
3. 建立 Qt 版 `RuntimeComposition`，只负责对象创建和依赖连接，不写业务逻辑。
4. 建立 `EventRegistration`，集中维护事件消费者订阅关系。
5. 保留现有主窗口和导航骨架，让 UI 壳在迁移期间始终可运行。

验收标准：

- Qt 项目可以构建和启动。
- Dashboard、Realtime Tags、Trend、Alarm Center、History、Measurement Map、Logs & Settings 仍能导航。
- 组合根文件存在，但不包含具体业务算法。

## 8. 高风险点和处理建议

### 8.1 C# async 到 Qt 线程模型

C# 使用 `Task`、`IAsyncEnumerable`、`CancellationToken` 和 `Channel`。Qt 版不能在 UI 线程中跑采集循环。建议：

- 采集主循环放入独立 `QThread` 或 worker object。
- UI 更新统一通过 queued connection 回主线程。
- Worker 停止使用原子 stop flag，并在退出时 flush 队列。
- SQLite 每个线程独立连接，禁止跨线程复用同一个 `QSqlDatabase`。

### 8.2 时间和 ticks

C# SQLite 中使用 `timestamp_utc_ticks`。Qt 默认毫秒时间戳和 C# ticks 不同。建议：

- 领域层定义统一 UTC 时间工具。
- SQLite 写入继续使用兼容 C# ticks，方便数据和测试对齐。
- UI 显示时再转本地时间。

### 8.3 领域层不要变成 Qt 对象层

如果把所有领域对象都做成 `QObject`，会造成复制困难、线程归属复杂、测试成本上升。建议：

- `domain` 和大部分 `application dto` 使用普通 C++ 值对象。
- 只有 UI ViewModel、Controller、Worker 外壳使用 `QObject`。

### 8.4 图表库选择

ScottPlot 没有直接等价的 Qt 官方控件。建议：

- 若追求工程速度，使用 QCustomPlot 或 Qwt。
- 若尽量少引入第三方，先使用 Qt Charts。
- 热力图不依赖趋势图库，单独用 `QPainter` 自绘。

### 8.5 中文与编码

原项目包含中文 Tag 名、页面文案、文档和 CSV。建议：

- 源码、文档、CSV 默认使用 UTF-8。
- CSV 导出保留 BOM。
- Qt 翻译可继续使用 `.ts`，但第一阶段可先硬编码中文/英文文案，后续统一国际化。

## 12. 最终判断

这个项目不适合按“C# 文件逐个翻译为 C++ 文件”的方式迁移。正确路线是保持 Clean Architecture 和数据闭环不变，把外围技术替换为 Qt6/C++：

- WPF Binding 替换为 Qt signal/slot、`Q_PROPERTY` 或页面快照刷新。
- DispatcherTimer 替换为 `QTimer`。
- C# async runtime 替换为 `QThread`/worker/线程安全队列。
- Microsoft.Data.Sqlite 替换为 QtSql SQLite。
- ScottPlot 替换为 Qt 图表库或自绘控件。
- Serilog 替换为 Qt 日志或 spdlog，但业务日志继续入 SQLite。

迁移完成的核心标志不是页面长得完全一样，而是同样的模拟帧进入系统后，Qt 版能得到同样的 Tag 状态、警报生命周期、历史采样、矩阵分析、UI 快照和用户操作结果。

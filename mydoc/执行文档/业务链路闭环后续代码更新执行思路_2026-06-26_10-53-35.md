## 2. 闭环目标定义

后续代码更新的目标不是再补单个页面控件，而是把当前 Qt 工程中已经迁移的运行时、事件、队列、SQLite 和 UI 快照真正接成同一条业务链路。

目标闭环如下：

```text
SimulatorDataSource
  -> MonitoringRuntimeService
  -> DataSourceHealthMonitor
  -> EventBus handlers
  -> TagService / MeasurementMapService / HistorySampleQueue / AlarmEventQueue / OperationLogQueue
  -> BatchPersistWorker
  -> SQLite repositories
  -> RuntimeUiSnapshotProvider / application query services
  -> MainWindow
  -> Dashboard / Realtime Tags / Trend / Alarm Center / History / Measurement Map / Logs & Settings
  -> UI commands
  -> application use cases
  -> runtime / repositories / configuration store
```

闭环完成后，主窗口不再依赖演示型 `UiSnapshotProvider` 自行造帧；真实模拟器帧应经过 runtime、事件消费者、缓存、队列、Worker 和 SQLite，再被 UI 以统一快照和查询服务消费。

## 3. 当前状态判断

### 3.1 已具备的基础能力

当前工程已经具备以下可复用能力：

- `SimulatorDataSource` 已实现 `IRawFrameSource`，可以作为真实 runtime 的默认数据源。
- `MonitoringRuntimeService` 已具备 RawFrame 采集、数据清洗、警报评估、Offline 超时发布和确认警报事件发布能力。
- `EventBus` 已支持订阅描述和真实 handler 注册，并支持 `Critical` / `Isolated` 失败策略。
- `RuntimeEventConsumers` 已具备 Tag 缓存、矩阵缓存、历史入队、警报入队、警报操作日志和数据源健康日志消费者。
- `HistorySampleQueue`、`AlarmEventQueue`、`OperationLogQueue` 和 `BatchPersistWorker` 已具备批量处理基础。
- SQLite 仓储已覆盖历史、警报、操作日志和配置；历史、警报、日志均已有分页查询模型或接口。
- UI 七个页面已经存在，并已补齐趋势预览、分页控件、取消查询状态和日志筛选等可见能力。

### 3.2 尚未闭环的关键断点

当前断点集中在接入层，而不是底层算法：

1. `RuntimeComposition` 当前主要初始化 SQLite 仓储和事件订阅描述，尚未创建完整 runtime 对象图。
2. `EventRegistration::registerApplicationEvents()` 只登记订阅描述；真实消费者实例尚未通过 `registerDefaultRuntimeConsumers()` 绑定到 `EventBus`。
3. `MainWindow` 当前自行创建 `UiSnapshotProvider`，该 provider 内部生成演示帧、维护内存历史和内存日志，未消费真实 runtime 产出的缓存、队列和 SQLite。
4. `AcquisitionRuntimeController` 当前 stop 时会 flush 历史，但 persistence runtime 何时启动、何时随应用退出停止还需要应用级 host 明确管理。
5. Alarm Center、History、Logs & Settings 的查询和保存入口仍然以快照内存数据为主，尚未通过应用层查询/设置服务接入 SQLite。
6. 配置保存后还没有统一更新 `RuntimeOptionsStore`、`AlarmService`、历史采样消费者和 SQLite，存在“页面保存了，但 runtime 未完全同步”的风险。

## 4. 后续设计原则

1. UI 不直接操作设备、模拟器、SQLite 表或仓储实现；UI 只调用 Application 层门面、快照源、查询服务或用例。
2. `RuntimeComposition` 只负责装配对象图和生命周期，不写数据清洗、警报、查询或 UI 业务逻辑。
3. 真实 runtime 和演示 provider 不能长期混用同一状态源；切换到闭环链路后，主窗口的 Start/Stop、确认警报、保存设置都必须走真实应用服务。
4. 页面刷新仍采用统一快照，不把采集线程的事件逐条推到 Widget。
5. 查询类交互允许由页面触发，但查询执行应在应用层服务中完成，仓储调用集中在服务或组合根注入对象中。
6. 采集线程、Worker 线程和 UI 线程之间只能传值对象或线程安全快照，避免 UI 读取正在被采集线程修改的容器。
7. SQLite 继续遵守每线程独立 connection name、UTC 时间、事务批量写入和 schema version 5 的契约。
8. 后续每一步都要保持可构建、可测试、可回退，避免一次性大改 MainWindow、RuntimeComposition 和所有页面。

## 5. 推荐新增或调整的应用层边界

后续写代码时建议先补清晰边界，再接 UI：

| 边界 | 建议职责 | 说明 |
| --- | --- | --- |
| `ApplicationRuntimeHost` 或等价对象 | 管理应用启动后的持久化 Worker、历史保留任务、退出清理 | 对齐 WPF README 中“应用启动持久化，用户点击 Start 才采集”的模型 |
| `RuntimeUiSnapshotProvider` | 从真实 runtime 缓存、状态和查询服务构建 `UiSnapshot` | 替代当前演示型 `UiSnapshotProvider` 的主窗口数据源 |
| `RuntimeCommandFacade` 或等价门面 | 提供 start、stop、acknowledge、save options、save tag configs 等 UI 命令入口 | 避免 MainWindow 直接拿多个 service 和 repository |
| `HistoryQueryService` | 封装 `SQLiteHistoryRepository::query()` | 返回分页结果，页面不直接依赖 Infrastructure |
| `AlarmQueryService` | 封装 `SQLiteAlarmRepository::query()` 和 open/latest 查询 | 支持 Alarm Center 历史区分页、时间范围、状态和等级过滤 |
| `OperationLogQueryService` | 封装 `SQLiteOperationLogRepository::queryPage()` | 支持 Logs & Settings 查询条件 |
| `RuntimeSettingsService` | 负责 runtime settings/tag configs 的加载、验证、保存和运行时同步 | 同时更新 SQLite、内存 store、AlarmService 和历史采样配置 |

如果暂时不想新增过多类，也应至少在组合根中暴露一个面向 UI 的 facade，避免 `MainWindow` 同时依赖 runtime、仓储、队列和多个 service。

## 6. 后续代码更新步骤

### 步骤 0：建立变更前基线

执行内容：

1. 运行当前 `cmake --build`、`ctest`、`MonitorCoreTests.exe` 和短启动 smoke。
2. 保留当前演示 `UiSnapshotProvider` 作为临时 fallback，不在第一步删除。
3. 明确后续新增测试名称，避免只靠手工 UI 验证。

验收：

- 当前工程无编译错误。
- 现有 7 个核心测试仍全绿。
- 文档中的后续步骤可逐步执行，不要求一次性完成全部闭环。

### 步骤 1：在组合根装配真实 runtime 对象图

执行内容：

1. 在 `RuntimeComposition` 中创建并持有默认 Tag 定义、Tag source mappings 和运行时配置。
2. 启动时从 `SQLiteConfigurationRepository` 加载 runtime settings 和 tag configs；如果数据库为空，则使用 `MonitorRuntimeOptions` 和 `TagDefinitionCatalog` 默认值。
3. 创建核心服务实例：
   - `RuntimeOptionsStore`
   - `DataCleanPipeline`
   - `AlarmService`
   - `TagService`
   - `DashboardService`
   - `ChartDataService`
   - `MeasurementMapService`
   - `OperationLogService`
   - `DataSourceHealthMonitor`
   - `SimulatorDataSource`
   - `MonitoringRuntimeService`
4. 创建三个队列：
   - `HistorySampleQueue`
   - `AlarmEventQueue`
   - `OperationLogQueue`
5. 创建三个 `BatchPersistWorker`：
   - `History`：批量调用 `SQLiteHistoryRepository::append()`
   - `Alarm`：批量调用 `SQLiteAlarmRepository::append()`
   - `OperationLog`：批量调用 `SQLiteOperationLogRepository::append()`
6. 创建 `PersistenceRuntimeCoordinator`、`RuntimeLifecycleCoordinator` 和 `AcquisitionRuntimeController`。
7. 为 `RuntimeComposition` 增加只读 accessor 或 facade accessor，供 `MainWindow` 使用。

逻辑要求：

- `RuntimeComposition` 负责对象生命周期，不能把对象放在 `main.cpp` 临时局部变量后再传裸指针给长期运行线程。
- Worker 的 persist callback 只能捕获生命周期长于 Worker 的仓储对象。
- `MonitoringRuntimeService` 的 run function 由 `RuntimeLifecycleCoordinator` 持有，stop flag 仍由 coordinator 控制。

验收：

- 组合根初始化后可以看到真实 runtime 对象均非空。
- 不启动采集时，应用也能初始化 SQLite、仓储、队列和 Worker 对象。
- 构建通过。

### 步骤 2：绑定真实 EventBus handler

执行内容：

1. 保留 `EventRegistration::defaultSubscriptions()` 作为订阅清单和启动校验依据。
2. 在真实消费者创建完成后调用 `registerDefaultRuntimeConsumers()`，将消费者实例绑定到 `EventBus` handler。
3. 确认 `EventBus::publish()` 不再只是跳过无 handler 的事件，而是能驱动：
   - `RawFrameReceivedEvent -> MeasurementMapFrameConsumer`
   - `TagRuntimeStatesProducedEvent -> TagCacheConsumer`
   - `TagRuntimeStatesProducedEvent -> HistoryRuntimeStateConsumer`
   - `AlarmRaised/Updated/Recovered/Acknowledged -> AlarmEventConsumer`
   - `AlarmRaised/Updated/Recovered/Acknowledged -> AlarmOperationLogConsumer`
   - `DataSourceTimedOut/Recovered -> DataSourceHealthOperationLogConsumer`
4. 增加启动校验：订阅描述存在但 handler 未绑定时，测试应能暴露。

逻辑要求：

- 订阅描述负责“应该有什么”，handler 绑定负责“实际做什么”，两者都要保留。
- `TagCacheConsumer` 仍保持 Critical，Tag 缓存失败应阻断 runtime。
- 历史、警报、日志和健康日志消费者保持 Isolated，避免单个落队列失败直接拖垮采集主链路。

验收：

- 单帧进入 `MonitoringRuntimeService::processFrame()` 后，Tag cache 有数据，历史队列有符合采样策略的数据，矩阵缓存有最新帧。
- 警报触发、确认、恢复能进入警报队列和操作日志队列。

### 步骤 3：补应用级 host 管理持久化生命周期

执行内容：

1. 新增或等价实现 `ApplicationRuntimeHost`，职责包括：
   - 应用初始化后启动 `PersistenceRuntimeCoordinator`
   - 执行历史保留任务
   - 应用退出时 stop runtime、stop persistence，并触发最终 flush
2. 明确采集 Start/Stop 与持久化 Worker 的关系：
   - 应用启动：SQLite 和 persistence Worker 可以先运行
   - 用户点击 Start：启动采集 runtime
   - 用户点击 Stop：停止采集 runtime，并 flush 历史队列
   - 应用退出：停止采集、停止所有 Worker，并 flush 历史、警报和日志
3. 将 `main.cpp` 或 `MainWindow` 的退出流程接入 host 清理。

逻辑要求：

- 不要在每次 Start 时重复创建 Worker。
- Stop 采集不等于关闭数据库；停止采集后 UI 仍可查询历史、警报和日志。
- 应用退出必须覆盖警报队列和操作日志队列 flush，不能只 flush 历史。

验收：

- 应用启动后数据库状态可显示 Running。
- 不启动采集也不应出现 Worker 崩溃。
- 关闭应用前队列数据可以被 flush。

### 步骤 4：实现真实 `RuntimeUiSnapshotProvider`

执行内容：

1. 保留 `UiSnapshot` DTO 作为 UI 统一输入模型。
2. 新增真实快照 provider，从以下来源构建快照：
   - Shell：`RuntimeLifecycleCoordinator::status()`、`PersistenceRuntimeCoordinator::status()`、`DataSourceHealthMonitor::status()`、最后帧号、矩阵帧号、同步状态。
   - Dashboard：`DashboardService::buildSnapshot()` + `TagService::snapshot()` + `AlarmService::currentAlarms()`。
   - Realtime Tags / Trend：`TagService::snapshot()` 和 `ChartDataService`。
   - Alarm 当前列表：`AlarmService::currentAlarms()`。
   - Alarm 历史默认列表：`AlarmQueryService` 查询最近 N 条，或先用 `AlarmService::recentAlarmEvents()` 作为短期过渡。
   - History 默认样本：`HistoryQueryService` 查询当前默认范围，或仅在页面主动查询时填充。
   - Operation Logs：`OperationLogQueryService` 查询最近 N 条。
   - Measurement Map：`MeasurementMapService::latestSnapshot()`。
   - Runtime Options / Tag Configs：`RuntimeOptionsStore` 和配置服务当前快照。
3. 将当前演示型 `UiSnapshotProvider` 改名或标记为 demo fallback，避免后续误用。

逻辑要求：

- 快照构建只读，不应在 `refresh()` 中生成新模拟帧。
- 快照构建必须跨线程安全；如果现有 service 的快照读取不完整，应先补 cache mutex 或新增不可变 `RuntimeSnapshotStore`。
- Shell 的 `databaseConnected` 应来自 persistence/SQLite 初始化状态，不再简单使用 `databaseReady()`。

验收：

- 未启动采集时，UI 显示 Idle/Stopped，但数据库状态正常。
- 启动采集后，Dashboard、Realtime Tags、Trend、Measurement Map 能从真实 runtime 缓存刷新。
- 停止采集后，UI 保留最后快照或显示 Stale，不再继续生成演示帧。

### 步骤 5：让 MainWindow 使用真实 facade

执行内容：

1. 修改 `MainWindow` 构造方式，让它接收 `RuntimeComposition` 暴露的 UI facade 或真实快照 provider。
2. 替换主窗口中的演示调用：
   - `startMonitoring()` 调用 `AcquisitionRuntimeController::start()` 或 facade `start()`
   - `stopMonitoring()` 调用 `AcquisitionRuntimeController::stop()` 或 facade `stop()`
   - `refreshShellClock()` 调用真实 `RuntimeUiSnapshotProvider::refresh()`
   - `acknowledgeAlarm()` 调用 `MonitoringRuntimeService::acknowledgeAlarm()` 或 `AcknowledgeAlarmUseCase`
3. 保留 `QTimer` 驱动 UI 刷新，刷新周期来自真实 runtime options。
4. 将 Start/Stop 结果、错误信息和 Faulted 状态显示到 Shell 或操作日志。

逻辑要求：

- `MainWindow` 不应自行创建业务 provider。
- UI 按钮只表达用户意图，实际状态以 runtime status 为准。
- Acknowledge 失败不能静默；需要日志或状态提示。

验收：

- 点击 Start 后真实 runtime 线程运行。
- 点击 Stop 后 runtime 线程停止，历史队列 flush。
- 确认警报后，当前警报状态改变，并最终写入警报事件和操作日志。

### 步骤 6：接入页面查询服务

执行内容：

1. Alarm Center：
   - 当前警报仍来自快照。
   - 历史区域改为调用 `AlarmQueryService`。
   - 支持时间范围、Tag、级别、状态、page/pageSize。
2. History：
   - 查询按钮调用 `HistoryQueryService`。
   - 支持 Tag、时间范围、page/pageSize、取消查询。
   - CSV 导出使用当前查询结果或全量匹配结果，不能只导出当前内存快照。
3. Logs & Settings：
   - 日志表调用 `OperationLogQueryService`。
   - 支持等级、category/action 文本、时间范围、page/pageSize。
4. 页面与服务之间建议通过 MainWindow/controller 连接信号槽，避免页面直接持有仓储。

逻辑要求：

- 查询条件使用 UI 本地时间时，进入仓储前必须转换为 UTC。
- 取消查询先可实现为忽略过期结果；后续再补真正异步取消。
- 查询结果应带 `totalCount`，页面页码按钮以 `hasPreviousPage()` / `hasNextPage()` 判断。

验收：

- 新产生的历史、警报和日志经过 Worker flush 后可以被页面查询到。
- 翻页不会越界，空结果页能稳定显示。
- CSV 导出使用 UTF-8 BOM，字段转义不回退。

## 8. 逻辑一致性检查清单

每次后续代码更新后，应检查以下一致性：

- 同一帧数据只能由一个 runtime 入口生成，不能同时由 `SimulatorDataSource` 和演示 `UiSnapshotProvider` 各生成一套数据。
- Start/Stop 状态来自 `RuntimeLifecycleCoordinator`，不能只看按钮状态。
- 数据库状态来自 SQLite/persistence runtime，不能只看数据库文件是否存在。
- Dashboard、Trend、Measurement Map 的快照应来自同一条 runtime 缓存链路，避免矩阵帧号和 Tag 帧号长期错位。
- 警报确认必须经过 `MonitoringRuntimeService::acknowledgeAlarm()` 或等价用例，并发布 `AlarmAcknowledgedEvent`。
- 历史、警报、日志写入必须经过队列和 Worker；UI 查询走只读查询服务。
- 设置保存必须同时落 SQLite 和更新运行时可见状态；不能只更新页面表格。
- UTC 时间进入仓储，本地时间只用于 UI 展示和用户输入转换。
- Worker 退出必须 flush，应用退出不能丢历史、警报和操作日志。

## 9. 不建议的实现方式

1. 不建议让页面直接 new SQLite 仓储或直接执行 SQL。
2. 不建议在 `MainWindow::refreshShellClock()` 中调用模拟器生成帧。
3. 不建议把 `RuntimeComposition` 写成业务流程脚本；组合根只装配对象。
4. 不建议为了快速显示数据绕过 `EventBus`，否则历史、警报和日志会再次断链。
5. 不建议删除现有演示 provider 后再一次性大改 UI；应先引入真实 provider 并切换主窗口，验证稳定后再清理。

## 10. 闭环完成验收标准

当以下条件全部满足时，可以认为业务链路闭环完成：

1. 应用启动后初始化 SQLite、配置仓储、事件 handler、队列、Worker 和真实 runtime 对象图。
2. 点击 Start 后，`SimulatorDataSource` 以默认 500 ms 周期供帧，UI 以默认 1000 ms 周期刷新。
3. RawFrame 能经过 `MonitoringRuntimeService`、`DataCleanPipeline`、`AlarmService` 和 `EventBus`，更新 Tag、矩阵、历史队列、警报队列和日志队列。
4. Worker 能把历史、警报和操作日志批量写入 SQLite。
5. Dashboard、Realtime Tags、Trend 和 Measurement Map 显示真实 runtime 数据，不再由 UI provider 造帧。
6. Alarm Center 能确认真实警报，确认事件和操作日志能被 SQLite 查询。
7. History、Alarm History 和 Operation Logs 查询来自 SQLite，支持分页。
8. Logs & Settings 保存的运行参数和 Tag 配置能落 SQLite，并能在重启后恢复。
9. Stop 采集后 UI 保留最后快照或显示 Stale，历史队列执行 flush。
10. 应用退出时 runtime 和 Worker 都能停止并 flush，无线程悬挂、无初始化崩溃。
